//   - scrollEditorWithPreview : preview scroll drives the editor
//   - refreshInterval         : ms debounce before re-render (try 0 and 500)
//   Toggle these per scenario as instructed.
//
// STICKY SCROLL: turn OFF `editor.stickyScroll.enabled` while comparing the top
// line of the editor with the top of the preview. Sticky scroll overlays the
// ancestor headings on top of the editor viewport, hiding the first lines, so
// the *visible* top line looks a few lines off from the synced one even though
// the mapping is correct.
//
// SMOOTH SCROLLING: the editor can only be scrolled to whole lines (no pixel
// API), so editor-follows-preview advances one line at a time. Turn ON
// `editor.smoothScrolling` so VS Code animates the transition between those
// positions and the follow looks gradual rather than stepwise.

== Scenario A — Typing in math far down should NOT jump the preview (#169, #709)

TEST: Scroll BOTH editor and preview down to this section. Put the caret in the
block equation below and type a few characters (e.g. add `+ 1`), then delete
them. Keep watching the preview viewport.

[stem]
++++
f(x) = \int_{-\infty}^{\infty} \hat{f}(\xi)\, e^{2 \pi i \xi x} \, d\xi
++++

EXPECTED: the equation re-typesets in place; the preview stays put (the line you
are editing does not fly to the top, the cursor area stays visible). Equations
ABOVE this point do NOT flash/re-render.

REGRESSION (old build): every keystroke replaced the whole webview, re-rendered
*all* MathJax from scratch, and the preview jumped — usually slamming the edited
block to the top of the view.

== Scenario B — Editing a table with math cells (#169, #873)

TEST: With the caret inside a cell below, edit a cell's expression. The cell (and
the whole table) changes height as MathJax re-lays-out.

[cols="1,1,1",options="header"]
|===
| Expression | Derivative | Value at 1

| stem:[x^2]            | stem:[2x]                 | stem:[2]
| stem:[\sin(x)]        | stem:[\cos(x)]            | stem:[\cos 1]
| stem:[e^{x}]          | stem:[e^{x}]              | stem:[e]
| stem:[\ln(x)]         | stem:[1/x]                | stem:[1]
| stem:[\int_0^1 x\,dx] | stem:[\tfrac{1}{2}x^2]    | stem:[\tfrac{1}{2}]

EXPECTED: only the edited cell re-typesets; the preview viewport is re-anchored
so the table does not jump away from you as its height changes.

REGRESSION: the table height change + full reload made the preview bounce, and
the offset around tables was wrong (#873).

== Scenario C — Saving with refreshInterval = 0 (#776)

TEST: Set `asciidoc.preview.refreshInterval` to `0` AND enable
`scrollPreviewWithEditor`. Scroll here, put the caret in the paragraph below,
type something, then press Ctrl/Cmd+S to save.

This paragraph is the save target. Edit me, then save and watch the viewport.

EXPECTED: saving re-renders but does NOT yank the preview to the caret or to the
top; your scroll position is preserved.

REGRESSION: with refreshInterval 0, live edits did not refresh, but *saving*
forced a full reload + scroll-to-cursor that fought your current position (#776).

== Scenario D — Scroll feedback loop, both directions enabled (#638)

TEST: Enable BOTH `scrollPreviewWithEditor` and `scrollEditorWithPreview`.
Slowly drag-scroll the PREVIEW with the mouse, then slowly scroll the EDITOR.

EXPECTED: smooth tracking. The content does not vibrate / bounce back and forth
between two positions.

REGRESSION: editor → preview → editor echo made the view oscillate (#638). The
re-anchoring scrolls we trigger are now flagged so they do not echo back.
